# Pulse LIVE/TEST Runtime Simulation

This document explains the TEST runtime channel for simulating multi-actor supervisor workflows without Telegram.

## Architecture

- Runtime mode switch:
  - `PULSE_RUNTIME_MODE=LIVE|TEST` (default: `LIVE`)
- Transport behavior:
  - `LIVE`: existing Telegram polling + send behavior.
  - `TEST`: no Telegram polling/sends. Inputs come from `Test_Inbox`; outputs are persisted to `Test_Outbox`/`Test_Attachments`.
- Shared business logic:
  - Existing handlers are reused (`fallback_text`, callback router, production handlers).
  - TEST runtime uses a fake Telegram update/context layer and routes into the same handlers.
- State persistence:
  - Per `session_id + actor_user_id` context is snapshotted in `Test_UserContext`.

## Test Tables

Created by `scripts/grist/setup_test_runtime_doc.py`:

- `Test_Inbox`: simulated user actions.
- `Test_Outbox`: message/notification payloads generated by handlers.
- `Test_UserContext`: persisted context snapshots.
- `Test_Attachments`: metadata for `reply_document`/`send_document`.
- `Test_RunLog`: processing and error log.

## Required Environment Variables

- `PULSE_RUNTIME_MODE=TEST`
- `PULSE_TEST_DOC_ID=<your_test_doc_id>`
- `PULSE_TEST_API_KEY=<api_key>` (optional if `PULSE_API_KEY` already points to a valid key)
- `PULSE_TEST_POLL_INTERVAL_SECONDS=30` (default set to 30)

Recommended safety:

- `PULSE_TEST_ALLOW_PROD_WRITES=false` (default)

When `PULSE_RUNTIME_MODE=TEST` and `PULSE_TEST_ALLOW_PROD_WRITES=false`, writes are blocked for all docs except `PULSE_TEST_DOC_ID`.

## Run in TEST Mode

```powershell
$env:PULSE_RUNTIME_MODE="TEST"
$env:PULSE_TEST_DOC_ID="tSFZW3ybtD46ug2q76iMML"
$env:PULSE_TEST_POLL_INTERVAL_SECONDS="30"
python -m pulse.main
```

Processor behavior:

- Polls `Test_Inbox` every 30s for `processed=false`.
- Routes each row:
  - `input_type=text`: text handler path
  - `input_type=callback`: callback router path
- Marks row processed and logs success/failure in `Test_RunLog`.

## Simulating Actions

Insert rows into `Test_Inbox`:

- `session_id`: workflow/test session key
- `actor_user_id`: Telegram ID of actor from Pulse Users table
- `input_type`: `text` or `callback`
- `payload`: message text or callback data
- `processed`: `false`

Examples:

```text
session_id=s-1001, actor_user_id=123456, input_type=text, payload=/start
session_id=s-1001, actor_user_id=123456, input_type=text, payload=New Production Batch
session_id=s-1001, actor_user_id=555000, input_type=callback, payload=prodappr|approve|42
session_id=s-1001, actor_user_id=777111, input_type=callback, payload=prodsv|schedule|42
```

For callback simulation, copy callback payload from `Test_Outbox.buttons_json`.

### CLI helper (recommended)


Notes:

- Requires `PULSE_RUNTIME_MODE=TEST`.
- Writes to `Test_Inbox` in `PULSE_TEST_DOC_ID`.
- Output prints inserted inbox row id.

## End-to-End Walkthrough (Batch -> Approval -> Schedule -> Handoff Complete)

This runbook uses your real Pulse/Costing master data (read-only in TEST mode) and test transport tables in `PULSE_TEST_DOC_ID`.

### 1. Set environment for TEST

Ensure these are set:

```powershell
$env:PULSE_RUNTIME_MODE="TEST"
$env:PULSE_TEST_DOC_ID="tSFZW3ybtD46ug2q76iMML"
$env:PULSE_TEST_POLL_INTERVAL_SECONDS="30"
$env:PULSE_TEST_ALLOW_PROD_WRITES="false"
```

Start runtime:

```powershell
python -m pulse.main
```

### 2. Pick actor Telegram IDs from Pulse `Users`

Use Pulse `Users.Telegram_ID` values for:

- Production Supervisor (creates batch)
- Production Manager (approves)
- Current-stage supervisor(s) for downstream MS actions
- Next-stage supervisor(s) for handoff confirmation

### 3. Create one session id

Use one `session_id` for this full scenario, for example:

- `sim-e2e-001`

### 4. Insert `/start` for supervisor and manager in `Test_Inbox`

Add two rows:

```text
session_id=sim-e2e-001, actor_user_id=<supervisor_tg_id>, input_type=text, payload=/start, processed=false
session_id=sim-e2e-001, actor_user_id=<manager_tg_id>, input_type=text, payload=/start, processed=false
```

Wait one poll cycle (30s) and verify new menu messages in `Test_Outbox`.

Equivalent helper commands:

```powershell
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <supervisor_tg_id> --role Production_Supervisor --text "/start"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <manager_tg_id> --role Production_Manager --text "/start"
```

### 5. Supervisor creates batch

Insert `Test_Inbox` rows for the supervisor following prompts in `Test_Outbox.message_text`.

Typical sequence:

1. Menu action that starts batch creation (label depends on your permissions/menu config).
2. Batch mode:
   - `By Product Model` or `By Product Part`
3. Model selection:
   - numeric text like `1`
4. If by part:
   - part numbers like `1,3`
   - `Yes` / `No` for more parts
5. Quantity:
   - e.g. `100`
6. Batch type:
   - `New Complete Batch (M-C-S)` or `MS Only` etc.
7. Confirm:
   - `Yes`

Example rows:

```text
session_id=sim-e2e-001, actor_user_id=<supervisor_tg_id>, input_type=text, payload=<menu label for new production batch>, processed=false
session_id=sim-e2e-001, actor_user_id=<supervisor_tg_id>, input_type=text, payload=By Product Model, processed=false
session_id=sim-e2e-001, actor_user_id=<supervisor_tg_id>, input_type=text, payload=1, processed=false
session_id=sim-e2e-001, actor_user_id=<supervisor_tg_id>, input_type=text, payload=100, processed=false
session_id=sim-e2e-001, actor_user_id=<supervisor_tg_id>, input_type=text, payload=New Complete Batch (M-C-S), processed=false
session_id=sim-e2e-001, actor_user_id=<supervisor_tg_id>, input_type=text, payload=Yes, processed=false
```

Equivalent helper commands:

```powershell
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <supervisor_tg_id> --text "<menu label for new production batch>"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <supervisor_tg_id> --text "By Product Model"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <supervisor_tg_id> --text "1"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <supervisor_tg_id> --text "100"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <supervisor_tg_id> --text "New Complete Batch (M-C-S)"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <supervisor_tg_id> --text "Yes"
```

After creation, manager receives approval CTA in outbox with callback payload:

- `prodappr:open:<batch_id>`

### 6. Manager opens and approves

In `Test_Outbox`, find manager row with button callback `prodappr:open:<batch_id>`.  
Insert callback row:

```text
session_id=sim-e2e-001, actor_user_id=<manager_tg_id>, input_type=callback, payload=prodappr:open:<batch_id>, processed=false
```

Manager then receives `Yes` / `Reject` buttons with callback:

- Approve: `prodappr:approve:<batch_id>`
- Reject: `prodappr:reject:<batch_id>`

Approve row:

```text
session_id=sim-e2e-001, actor_user_id=<manager_tg_id>, input_type=callback, payload=prodappr:approve:<batch_id>, processed=false
```

Equivalent helper commands:

```powershell
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <manager_tg_id> --callback "prodappr:open:<batch_id>"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <manager_tg_id> --callback "prodappr:approve:<batch_id>"
```

Expected result:

- Batch status moves to approved/schedule-pending flow states.
- Downstream supervisor notifications appear in `Test_Outbox`.

### 7. Schedule the approved batch

For the authorized supervisor, trigger schedule using either menu flow or callback.

Callback pattern:

- `prodsv:schedule:<batch_id>`

Then supply date when prompted:

- `Today`, `Tomorrow`, or `YYYY-MM-DD`

Example:

```text
session_id=sim-e2e-001, actor_user_id=<stage_supervisor_tg_id>, input_type=callback, payload=prodsv:schedule:<batch_id>, processed=false
session_id=sim-e2e-001, actor_user_id=<stage_supervisor_tg_id>, input_type=text, payload=Today, processed=false
```

Equivalent helper commands:

```powershell
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <stage_supervisor_tg_id> --callback "prodsv:schedule:<batch_id>"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <stage_supervisor_tg_id> --text "Today"
```

Expected result:

- Master/MS scheduled date set.
- Outbox confirms scheduling and shows action menus.

### 8. Execute stage action and handoff confirmation

For each active MS row/stage:

1. Current-stage supervisor marks done:
   - callback `prodsv:done_row:<row_id>` or equivalent batch callback from outbox.
2. System moves row to `Done - Pending Confirmation`.
3. Next-stage supervisor confirms handoff:
   - callback `prodsv:confirm_row:<row_id>`

Example:

```text
session_id=sim-e2e-001, actor_user_id=<current_stage_tg_id>, input_type=callback, payload=prodsv:done_row:<row_id>, processed=false
session_id=sim-e2e-001, actor_user_id=<next_stage_tg_id>, input_type=callback, payload=prodsv:confirm_row:<row_id>, processed=false
```

Equivalent helper commands:

```powershell
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <current_stage_tg_id> --callback "prodsv:done_row:<row_id>"
python scripts/grist/push_test_inbox.py --session sim-e2e-001 --actor <next_stage_tg_id> --callback "prodsv:confirm_row:<row_id>"
```

Repeat for remaining rows/stages until the final stage is confirmed.

### 9. Verify completion and no cross-supervisor leakage

Check:

- `Test_Outbox` rows for each actor only contain intended jobs/actions.
- `buttons_json` callback payloads map to correct actor and stage.
- `Test_RunLog` has no errors.
- `Test_UserContext` stays isolated per `session_id + actor_user_id`.

To specifically validate your bug ("View All showing other supervisor jobs"), run the same menu path as each supervisor in separate inbox rows and compare their outbox payloads.

## Switch Back to LIVE Safely

```powershell
$env:PULSE_RUNTIME_MODE="LIVE"
Remove-Item Env:PULSE_TEST_DOC_ID -ErrorAction SilentlyContinue
python -m pulse.main
```

## Notes / Limitations

- TEST mode validates payload structure (text/buttons/callbacks), not Telegram device-specific rendering quirks.
- With write guards enabled, any flow that writes to non-test docs will be blocked by design.
